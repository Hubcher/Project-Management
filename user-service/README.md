# UserService

`user-service` отвечает за хранение профилей пользователей.

Это внутренний gRPC-сервис. Внешние клиенты не должны работать с ним напрямую: весь публичный доступ идет через `gateway`.

## Роль в системе

Сервис хранит пользовательские данные, которые не относятся к учетным данным и безопасности.

Здесь находятся:
- имя
- фамилия
- отчество
- дата рождения
- телефон
- отдел
- должность
- ссылка на аватар
- bio

Аутентификационные данные здесь не хранятся. Они живут в `auth-service`.

## Таблицы БД

### `users`

Основные поля:
- `id`
- `first_name`
- `last_name`
- `middle_name`
- `birth_date`
- `phone`
- `department`
- `position`
- `avatar_url`
- `bio`
- `created_at`
- `updated_at`

## Что умеет сервис

Полный CRUD по пользовательским профилям:
- создать профиль
- получить профиль по `id`
- получить список существующих пользовательских профилей
- обновить профиль
- удалить профиль

## Внутренний интерфейс

Сервис публикует gRPC API:
- `Ping`
- `CreateUser`
- `GetUserById`
- `ListUsers`
- `UpdateUser`
- `DeleteUser`

Контракт находится в:
- [`contracts/proto/user/user.proto`](../contracts/proto/user/user.proto)

## Как сервис используется в системе

Через `gateway`:
- обычный пользователь может получить и обновить свой профиль
- администратор может получить список пользователей
- администратор может удалить пользователя
- администратор может создать профиль пользователю
- создание профиля при саморегистрации происходит через orchestration `gateway -> auth-service + user-service`

Важно:
- правила доступа определяются в `gateway`
- сам `user-service` не является публичным REST API

## Конфигурация

Базовый файл:
- `user-service/configs/config.yaml`

Поддерживаемые override:

| YAML | Env | Значение |
| --- | --- | --- |
| `log_level` | `LOG_LEVEL` | уровень логирования |
| `address` | `USER_ADDRESS` | адрес gRPC-сервера |
| `db_address` | `DB_ADDRESS` | строка подключения к PostgreSQL |

## Запуск

Локально:

```bash
go run ./cmd -config configs/config.yaml
```

Через Docker Compose из корня репозитория:

```bash
docker compose up --build -d user-service
```

## Миграции

Миграции находятся в:
- `user-service/internal/adapters/db/postgres/migrations`

Текущая схема создается файлом:
- `000001_create_users.up.sql`

## Что важно помнить

- у сервиса нет внешнего REST API
- Swagger для него не ведется
- внешняя документация по пользователям находится только в `gateway`
- учетные данные и роли находятся не здесь, а в `auth-service`
